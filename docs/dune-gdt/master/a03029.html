Detailed Description

template<class OperatorImp, class DiscreteFunctionImp, TimeStepperMethods method = TimeStepperMethods::implicit_euler, XT::LA::Backends container_backend = XT::LA::default_sparse_backend>
class Dune::GDT::DiagonallyImplicitRungeKuttaTimeStepper< OperatorImp, DiscreteFunctionImp, method, container_backend >

Time stepper using diagonally implicit Runge Kutta methods.

Timestepper using diagonally implicit Runge Kutta methods to solve equations of the form u_t = r * L(u, t) where u is a discrete function, L an operator acting on u and r a scalar factor (e.g. -1). The specific Runge Kutta method can be chosen as the third template argument. If your desired Runge Kutta method is not contained in DiagonallyImplicitRungeKuttaMethods, choose TimeStepperMethods::diagonally_implicit_other and supply a DynamicMatrix< RangeFieldType > A and vectors (DynamicVector< RangeFieldType >) b and c in the constructor. Here, A, b and c form the butcher tableau (see https://en.wikipedia.org/wiki/List_of_Runge%E2%80%93Kutta_methods, A is composed of the coefficients a_{ij}, b of b_j and c of c_j). The default is a backward euler method.

Notation (the factor r is omitted for notational simplicity): For an s-stage method, {u}^{n+1} = {u}^n + dt {i=0}^{s-1} b_i {k}_i {k}_i = L({u}_i, t_i) t_i = t^n + dt c_i {u}_i = {u}^n + dt {j=0}^{s-1} a_{ij} {k}_j, To solve for the {u}_i, a Newton method with Armijo backtracking line search is used at each stage (if a_{ii} 0). The Newton update for each stage takes the form N({u}_l) {d}_l := (I - dt a_{ii} J_L({u}_l, t_i)) {d}_l = -{u}_l

  • {u}_{expl, i} + dt a_{ii} L({u}_l, t_i) =: -{res}({u}_l), where {u}_l is the current iterate, {u}_{expl, i} = {u}^n + dt {j=0}^{j-1} b_j {k}_j the explicit part of {u}_i, J_L the jacobian of the operator L with respect to {u} and {u}_{l+1} = {u}_l + {d}_l. The factor is found by an Armijo backtracking line search such that {res}({u}_l + * {d}_l) < {res}({u}^n) + D{res} {d}_l, where D{res} = -N is the jacobian of the residual.

Parameters

num_save_stepsNumber of time points that will be stored in solution if save_solution is true and/or visualized if visualize_solution is true. Default is size_t(-1), which will store all time steps.
save_solutionIf true, the discrete solution at num_save_steps equidistant time points will be stored in solution.
output_progressIf true, the current time and current time step length will be written to std::cout at num_save_steps equidistant time points
visualizeIf true, solution will be written to .vtp/.vtu files at num_save_steps equidistant time points
filename_prefixPrefix of .vtp/.vtu files for visualization
solOptional object that solution will be appended to instead of storing it in the time stepper.
Returns
estimated optimal time step length for next step
Note
If num_save_steps is specified (i.e. if it is not size_t(-1), the solution will be stored/visualized at exactly num_save_steps + 1 equidistant time points (including the initial time and t_end), even if the time step length has to be reduced to hit these time points.

Perform one time step and return the estimated optimal time step length for the next time step.

Parameters
dtTime step length.
max_dtMaximal allowed time step length in this time step.
Returns
Estimated optimal time step length for the next time step.
Note
For adaptive methods or if max_dt < dt, dt is just the initial time step length, the actual time step taken may be shorter. To get the actual time step length you need to compare current_time() before and after calling step().
This method has to increase the current time by the actual time step length taken, otherwise solve will never finish execution (if current_time is not increased) or give wrong results (if current time is increased by a wrong dt).
